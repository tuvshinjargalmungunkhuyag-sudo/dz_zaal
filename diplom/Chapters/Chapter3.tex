\section{Системийн шаардлага ба загварчлал}

\subsection{Функциональ болон функциональ бус шаардлага}

Системийн шаардлагыг Удиртгалд тогтоосон таван асуудалд тулгуурлан боловсруулав: утсаар захиалах хүндрэл, биечлэн очих зайлшгүй байдал, цагийн мэдээлэл ил тод бус байдал, захиалга цуцлах боломжгүй байдал, сануулга байхгүй байдал. Эдгээр асуудал бүрийг шийдвэрлэх функциональ шаардлагыг дараах байдлаар тодорхойлсон.

Хэрэглэгчийн функцийн хувьд: email хаяг болон нууц үгээр бүртгүүлэх, бүртгэлийн үед тухайн email рүү 6 оронтой баталгаажуулах код илгээгдэж, кодоо оруулсны дараа л нэвтрэх боломж нээгдэх; таван спорт байгууламжийн жагсаалтыг нүүр дэлгэцэд харах; тухайн заалны боломжтой болон хаагдсан цагийн хуваарийг харах; цаг захиалах, DBK-xxxxx форматын захиалгын код авах; идэвхтэй захиалгаа цуцлах; захиалгын түүхийг харах; захиалгаас 1 цагийн өмнө локал мэдэгдэл хүлээн авах; AI чатботтой спорт заалтай холбоотой асуулт асуух. Системийн автомат функцийн хувьд 15 минут тутамд cron ажил дуусгавар болсон захиалгыг \code{completed} төлөвт шилжүүлэх, байгууллагуудын тогтмол (давтагдах) захиалгыг \code{fixed\_bookings} цуглуулгаас уншиж долоо хоног бүр цагийн хуваарьт тусгах зэрэг функц орно.

Функциональ бус шаардлагыг хэмжигдэхүйц хэлбэрээр тодорхойлсон бөгөөд тэдгээрийг доорх хүснэгтэд нэгтгэв.

\section{Frontend хөгжүүлэлт}

\subsection{Дизайн систем ба навигаци}

Апп-ын дизайн нь говийн цөлийн дулаан тонийг илэрхийлсэн, нүдэнд ээлтэй цайвар дэвсгэр дээрх наранжин-хүрэн өнгөн схемтэй. Бежийн дэвсгэр (\code{\#EDE7DA}), наранжин (\code{\#D4700F}), гүн наранжин (\code{\#B85C08}), харанхуй хүрэн (\code{\#1A1208}) дөрвөн үндсэн өнгийг \code{Montserrat} фонттой хослуулав. Material Design 3-ийн \code{ThemeData} классаар дамжуулан апп-ын бүх widget-д нэгдсэн дизайн систем хэрэгжүүлсэн:

\begin{lstlisting}[language={}, caption={Material Design 3 дулаан говийн дизайн тохиргоо --- AppTheme класс}]
class AppTheme {
  static const Color primary       = Color(0xFFEDE7DA);
  static const Color secondary     = Color(0xFFD4700F);
  static const Color accent        = Color(0xFFB85C08);
  static const Color textPrimary   = Color(0xFF1A1208);

  static ThemeData get lightTheme => ThemeData(
    useMaterial3: true,
    fontFamily: 'Montserrat',
    brightness: Brightness.light,
    scaffoldBackgroundColor: primary,
    colorScheme: const ColorScheme.light(
      primary:   secondary,
      secondary: accent,
    ),
  );
}
\end{lstlisting}

Апп-ын үндсэн навигаци нь \code{BottomNavigationBar} ашиглан 4 tab-тай: Нүүр (Home), Захиалгууд (Bookings), Чат (Chat), Профайл (Profile). Идэвхтэй tab-ийн индексийг root \code{StatefulWidget}-ийн \code{setState}-д хадгалдаг. Навигацийн бүтэц болон дэлгэцүүдийн удирдлагыг доорх кодоор хэрэгжүүлсэн:

\begin{lstlisting}[language={}, caption={BottomNavigationBar --- 4 tab навигаци, MainScreen класс}]
class _MainScreenState extends State<MainScreen> {
  int _selectedIndex = 0;
  final List<Widget> _screens = const [
    HomeScreen(), BookingsScreen(),
    ChatScreen(), ProfileScreen(),
  ];

  @override
  Widget build(BuildContext context) {
    return Scaffold(
      body: _screens[_selectedIndex],
      bottomNavigationBar: BottomNavigationBar(
        currentIndex: _selectedIndex,
        onTap: (index) => setState(() => _selectedIndex = index),
        items: const [
          BottomNavigationBarItem(icon: Icon(Icons.home), label: 'Home'),
          BottomNavigationBarItem(icon: Icon(Icons.calendar_today), label: 'Bookings'),
          BottomNavigationBarItem(icon: Icon(Icons.chat), label: 'Chat'),
          BottomNavigationBarItem(icon: Icon(Icons.person), label: 'Profile'),
        ],
      ),
    );
  }
}
\end{lstlisting}

Flutter апп нь давхаргат бүтэцтэй: \code{lib/screens/} дэлгэцүүд, \code{lib/widgets/} дахин ашиглагдах widget-ууд, \code{lib/services/} Firebase болон API холболтын классууд, \code{lib/models/} өгөгдлийн загварууд, \code{lib/config.dart} тохиргооны нэг эх сурвалж. Апп-ын бүтэн архитектурыг доорх диаграммаар харуулав.

Future<void> _loadSlots(String venueId, String date) async {
  final bookSnap = await FirebaseFirestore.instance
      .collection('bookings')
      .where('venueId', isEqualTo: venueId)
      .where('date', isEqualTo: date)
      .where('status', whereIn: ['upcoming', 'active'])
      .get();

  final weekday = DateTime.parse(date).weekday % 7;
  final fixedSnap = await FirebaseFirestore.instance
      .collection('fixed_bookings')
      .where('venueId', isEqualTo: venueId)
      .where('dayOfWeek', isEqualTo: weekday)
      .where('isActive', isEqualTo: true)
      .get();

  final bookedSlots = {
    ...bookSnap.docs.map((d) => d['timeSlot'] as String),
    ...fixedSnap.docs.map((d) => d['timeSlot'] as String),
  };
  setState(() => _bookedSlots = bookedSlots);
}
\end{lstlisting}

AI чатботын архитектур нь хоёр давхар горимтой: Firebase Cloud Functions дамжуулан Anthropic Claude Haiku 4.5 (үндсэн горим), Node.js бэкэнд дамжуулан Groq Llama 3.1-8b (нөөц горим). Groq нөөц горимын endpoint нь \code{/api/chat} хаягаар POST хүсэлт хүлээн авч Groq SDK-аар Llama 3.1-8b загвараар хариулт авдаг. Claude алдаа гарсан тохиолдолд Flutter апп автоматаар энэ endpoint рүү хандана.

Бэкэнд нь 15 минут тутамд cron ажлыг ажиллуулж, эхлэх цаг нь өнгөрсөн боловч \code{upcoming} төлөвтэй захиалгуудыг \code{completed} болгон шинэчилдэг. Ингэснээр Firestore-д хэрэглэгчийн захиалгын мэдээлэл бодит байдалтай нийцсэн байдлаар шинэчлэгдэнэ.

\begin{lstlisting}[language={}, caption={Cron ажил --- 15 минут тутамд дууссан захиалга шинэчлэх}]
cron.schedule('*/15 * * * *', async () => {
  const now   = new Date();
  const today = now.toISOString().split('T')[0];
  const nowTime = now.getHours().toString().padStart(2,'0') + ':'
                + now.getMinutes().toString().padStart(2,'0');
  const snapshot = await admin.firestore()
    .collection('bookings')
    .where('status', '==', 'upcoming')
    .where('dateKey', '<=', today).get();
  const batch = admin.firestore().batch();
  snapshot.docs.forEach(doc => {
    const endTime = doc.data().timeSlotEnd;
    if (doc.data().dateKey < today ||
        (doc.data().dateKey === today && endTime <= nowTime)) {
      batch.update(doc.ref, { status: 'completed' });
    }
  });
  await batch.commit();
});
\end{lstlisting}

\section{Өгөгдлийн сан}

Cloud Firestore-д дөрвөн collection ашигласан. Спорт заалнуудын мэдээлэл Firestore-д хадгалагддаггүй --- кодод шууд тодорхойлсон байдлаар тавьж, Firestore-ийн уншилтын тоог хэмнэдэг.

\keyword{users} collection нь хэрэглэгч бүрийн мэдээллийг хадгалдаг бөгөөд document ID нь Firebase Auth-ийн UID байна. uid, name, email, emailVerified (boolean), createdAt (timestamp) талбаруудтай. Хэрэглэгч бүртгүүлэх үед \code{emailVerified: false}, баталгаажсаны дараа \code{emailVerified: true} болдог.

\keyword{bookings} collection нь хэрэглэгчдийн бүх захиалгыг хадгалдаг. DBK-xxxxx форматын захиалгын код (code), хэрэглэгчийн UID (userId), email (userEmail), нэр (userName), заалны ID (venueId), огноо (date, dateKey), цагийн эхлэл/төгсгөл (timeSlot, timeSlotEnd), талбайн төрөл (courtType: full|half), үнэ (price), төлөв (status: upcoming|cancelled|completed) талбаруудтай. dateKey нь YYYY-MM-DD форматаар хадгалагддаг бөгөөд Firestore-ийн шүүлтийн хурдыг нэмэгдүүлнэ.

\keyword{fixed\_bookings} collection нь байгууллагуудын долоо хоногт давтагдах тогтмол цагийг хадгалдаг. venueId (заалны ID), organizationName (байгууллагын нэр), dayOfWeek (долоо хоногийн өдөр: 0=Ням---6=Бямба), timeSlot (цагийн үе), isActive (идэвхтэй эсэх) талбаруудтай. Энэ collection-оос уншсан өгөгдлөөр засгийн газар, компаниудын тогтмол цагийг цагийн хуваарьт тусгаж, хэрэглэгч тэр цагийг захиалах боломжгүй болгодог.

\keyword{email\_verifications} collection нь бүртгэлийн баталгаажуулах кодыг түр хадгалдаг. Document ID нь Firebase Auth-ийн UID байна. code (6 оронтой тоон код), email, expiresAt (10 минутын дараа дуусах), createdAt талбаруудтай. Баталгаажуулалт амжилттай болсны дараа тухайн document автоматаар устгагддаг.
